以下是根据前端设计文档及项目需求（AI英语口语陪练）编写的**后端设计说明**，采用 **Spring Boot** 作为核心框架，并融合端到端语音大模型（豆包 Realtime API / GPT-4o Realtime）的实现思路。

---

# LingoAI 后端设计说明

## 1. 项目概述

LingoAI 后端为前端提供 REST API 服务，负责用户会话管理、场景配置、对话记录存储、发音评测集成、课后总结生成等非实时业务。**实时语音对话**由前端直连端到端语音大模型（如豆包 Realtime API）完成，后端不直接处理音频流，以降低延迟和开发复杂度。

### 1.1 技术栈

| 分类 | 技术 | 版本 | 说明 |
|------|------|------|------|
| 核心框架 | Spring Boot | 3.2.x | 主后端框架 |
| Web 框架 | Spring Web | - | REST API |
| WebSocket | Spring WebSocket | - | 可选，用于代理模式或推送通知 |
| 数据库 | PostgreSQL | 15+ | 关系型数据库 |
| 缓存 | Redis | 7.x | 会话缓存、临时数据 |
| ORM | Spring Data JPA | - | 数据库操作 |
| 认证 | JWT | 0.12.x | 用户认证（可选） |
| HTTP 客户端 | RestClient / WebClient | - | 调用第三方 API |
| 构建工具 | Maven / Gradle | - | 项目构建 |

## 3. API 设计

### 3.1 场景管理

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/scenarios` | 获取所有场景列表（含欢迎语） |
| GET | `/api/scenarios/{id}` | 获取单个场景详情（含问题列表） |

**响应示例**（`GET /api/scenarios`）：
```json
{
  "code": 200,
  "message": "success",
  "data": [
    {
      "id": "interview",
      "title": "面试准备",
      "tag": "技术面试",
      "emoji": "💼",
      "difficulty": 3,
      "description": "模拟外企技术岗面试",
      "welcomeMessage": "Hello, I'm the interviewer. Please introduce yourself.",
      "welcomeTranslation": "你好，我是面试官。请做自我介绍。"
    }
  ]
}
```

### 3.2 对话会话管理

| 方法 | 路径 | 描述 |
|------|------|------|
| POST | `/api/conversations` | 开始新会话（返回 sessionId） |
| POST | `/api/conversations/{sessionId}/messages` | 提交用户回答（文本） |
| GET | `/api/conversations/{sessionId}` | 获取完整对话历史 |
| DELETE | `/api/conversations/{sessionId}` | 结束会话，触发总结生成 |

**开始会话请求**：
```json
{
  "scenarioId": "interview",
  "userId": "user123"   // 可选，未登录可匿名
}
```

**响应**：
```json
{
  "code": 200,
  "data": {
    "sessionId": "conv_123456",
    "startTime": "2026-06-05T10:00:00Z"
  }
}
```

### 3.3 发送消息（用户回答）

**请求**：
```json
{
  "text": "I have three years of experience in full-stack development.",
  "useVoice": false   // 是否为语音输入（影响评测侧重点）
}
```

**响应**（包含 AI 回复 + 评测结果）：
```json
{
  "code": 200,
  "data": {
    "aiMessage": {
      "role": "ai",
      "text": "That's impressive! Could you tell me about a challenging project you worked on?",
      "translation": "太棒了！能说说你做过的一个有挑战的项目吗？"
    },
    "rating": {
      "accuracy": 85,
      "fluency": 90,
      "grammar": 82,
      "overall": 88,
      "feedbackSummary": "发音清晰，但注意'experience'的重音"
    },
    "grammarFeedback": {
      "original": "I have three years of experience...",
      "suggested": "I have three years of experience in...",
      "title": "表达优化",
      "explanation": "你的表达已经很好了，可以更简洁"
    }
  }
}
```

> **注意**：实际场景中，AI 回复由前端直连的语音大模型生成。后端不直接返回 `aiMessage`，而是负责存储前端传来的用户消息和 AI 回复（通过回调或异步方式）。为简化，初期可让前端同时调用后端 API 存储对话记录，AI 回复由前端自己显示。

**更符合端到端模式的实现**：
- 前端与语音大模型完成一轮对话后，调用后端 API `POST /api/conversations/{sessionId}/record` 上传（用户文本、AI 文本、评测结果），后端仅做存储和总结生成。

### 3.4 发音评测（异步/同步）

由于发音评测可由前端直接调用第三方 API 并上报结果，后端也可以代理评测请求以统一管理密钥。

| 方法 | 路径 | 描述 |
|------|------|------|
| POST | `/api/pronunciation/evaluate` | 接收音频 Base64，调用评测服务返回结果 |

**请求**：
```json
{
  "audioBase64": "data:audio/webm;base64,...",
  "referenceText": "I have three years of experience"
}
```

**响应**：同前述 `rating` 结构。

### 3.5 练习总结

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/api/conversations/{sessionId}/summary` | 获取总结数据（雷达图、错误列表、建议） |

**响应示例**：
```json
{
  "code": 200,
  "data": {
    "overallScore": 86,
    "dimensions": {
      "pronunciation": 85,
      "grammar": 78,
      "fluency": 90,
      "vocabulary": 82,
      "interactivity": 88
    },
    "errors": [
      {
        "original": "I have do a project",
        "corrected": "I have done a project",
        "type": "verb tense"
      }
    ],
    "suggestions": [
      "练习 /θ/ 音，如 'think', 'thank'",
      "注意现在完成时结构：have/has + past participle"
    ]
  }
}
```

### 4.2 主要表结构

#### `scenarios`（场景表）
```sql
CREATE TABLE scenarios (
    id VARCHAR(50) PRIMARY KEY,
    title VARCHAR(100),
    tag VARCHAR(50),
    emoji VARCHAR(10),
    difficulty INT,
    description TEXT,
    welcome_message TEXT,
    welcome_translation TEXT
);
```

#### `conversations`（会话表）
```sql
CREATE TABLE conversations (
    id VARCHAR(50) PRIMARY KEY,
    user_id VARCHAR(50),          -- 可为空（匿名用户）
    scenario_id VARCHAR(50),
    start_time TIMESTAMP,
    end_time TIMESTAMP,
    overall_score INT
);
```

#### `messages`（消息表）
```sql
CREATE TABLE messages (
    id VARCHAR(50) PRIMARY KEY,
    conversation_id VARCHAR(50),
    role VARCHAR(10),            -- 'user' or 'ai'
    text TEXT,
    translation TEXT,
    pronunciation_score INT,
    grammar_feedback JSONB,      -- 存储 { original, corrected, type }
    timestamp TIMESTAMP
);
```

#### `user_progress`（用户进度，可选）
```sql
CREATE TABLE user_progress (
    id SERIAL PRIMARY KEY,
    user_id VARCHAR(50),
    scenario_id VARCHAR(50),
    best_score INT,
    practice_count INT,
    last_practice TIMESTAMP
);
```

## 5. 核心业务流程实现

### 5.1 开始新会话

1. 前端调用 `POST /api/conversations`，携带 `scenarioId` 和可选的 `userId`。
2. 后端生成唯一 `sessionId`（UUID），存入 Redis（TTL 24h），同时初始化数据库会话记录。
3. 返回 `sessionId` 给前端。
4. 前端使用该 `sessionId` 后续存储消息。

### 5.2 用户提交回答（记录模式）

由于音频对话由前端直接与语音大模型交互，后端不参与实时对话。流程为：

1. 用户完成一轮与 AI 的语音对话后，前端获得：
   - 用户 ASR 文本
   - AI 回复文本（含翻译）
   - 发音评测结果（可选，也可由后端调用评测）
2. 前端调用 `POST /api/conversations/{sessionId}/messages` 将上述数据提交后端存储。
3. 后端将消息存入数据库，并触发可选的后处理：
   - 更新会话总体评分（累计）
   - 若消息中包含语法纠正，存入 grammar_feedback JSONB 字段。

### 5.3 发音评测集成

后端提供代理接口 `POST /api/pronunciation/evaluate`，内部调用第三方服务（驰声 CHIVOX、Azure Speech Assessment 或开源 Echoic）。流程：

1. 接收前端传来的音频 Base64 和参考文本。
2. 转换为第三方 API 所需格式（如 PCM 16kHz）。
3. 调用评测 API，解析返回的评分及音素级反馈。
4. 返回统一格式的评分给前端。

**示例代码（伪代码）**：
```java
public PronunciationScore evaluate(String audioBase64, String text) {
    // 解码音频
    byte[] audio = Base64.getDecoder().decode(audioBase64);
    // 调用驰声 API
    ChivoxResponse response = chivoxClient.evaluate(audio, text);
    // 映射为统一评分
    return new PronunciationScore(
        response.getAccuracy(),
        response.getFluency(),
        response.getGrammar(),
        response.getOverall(),
        response.getFeedback()
    );
}
```

### 5.4 练习总结生成

当用户点击“结束练习”或前端调用 `DELETE /api/conversations/{sessionId}` 时：

1. 后端查询该会话下所有消息及评测分数。
2. 计算各维度平均分（发音、语法、流利度、词汇量等）。
   - 词汇量可通过消息文本的词频、多样性估算。
   - 互动主动性可根据用户消息数量与 AI 消息比例计算。
3. 汇总所有 grammar_feedback 中的错误，去重后按类型归类。
4. 根据低分维度生成个性化建议（基于规则或 LLM 生成）。
5. 存储总结结果（可存 JSON 字段到 conversations 表）。
6. 返回总结数据给前端。

## 6. 与前端实时语音的集成方式

### 6.1 推荐方案：前端直连语音 AI + 后端仅存储

- **优点**：最低延迟，后端负载极小，开发快。
- **缺点**：语音大模型的 API Key 需暴露在前端（可通过后端临时 Token 解决）。

**Token 安全方案**：
1. 前端先调用后端 `POST /api/token` 获取临时访问凭证（如限时 1 小时）。
2. 前端使用该凭证连接豆包/OpenAI 的 WebSocket 服务（服务端需支持动态 Token）。
3. 若语音 AI 不支持动态 Token，则后端需做 WebSocket 代理（见备选方案）。

### 6.2 备选方案：后端代理 WebSocket

前端连接 `ws://backend:8080/voice`，后端建立与语音 AI 的 WebSocket 连接，做透明转发。这样 API Key 存放于后端，安全可控。

**实现要点**：
- 使用 `@EnableWebSocket` 和 `WebSocketHandler`。
- 每个客户端会话对应一个后端客户端会话，使用 `ConcurrentHashMap` 管理。
- 数据帧透传：二进制音频直接转发，文本消息按需处理（如记录日志）。

**代理模式的延迟增加 < 50ms**，在可接受范围。

## 7. 安全与配置

### 7.1 认证授权（可选 MVP 阶段可跳过）

- 使用 JWT，登录接口 `/api/auth/login` 返回 token。
- 后续请求在 `Authorization: Bearer <token>` 头中携带。
- 无用户时可分配匿名 sessionId。

### 7.2 CORS 配置

```java
@Configuration
public class CorsConfig {
    @Bean
    public WebMvcConfigurer corsConfigurer() {
        return new WebMvcConfigurer() {
            @Override
            public void addCorsMappings(CorsRegistry registry) {
                registry.addMapping("/api/**")
                        .allowedOrigins("http://localhost:5173")  // 前端开发地址
                        .allowedMethods("*");
            }
        };
    }
}
```

### 7.3 环境变量

| 变量名 | 说明 | 示例 |
|--------|------|------|
| `DB_URL` | PostgreSQL 连接 | `jdbc:postgresql://localhost:5432/lingoai` |
| `DB_USER` | 数据库用户名 | `postgres` |
| `DB_PASSWORD` | 数据库密码 | `***` |
| `REDIS_HOST` | Redis 地址 | `localhost` |
| `REDIS_PORT` | Redis 端口 | `6379` |
| `PRONUNCIATION_API_KEY` | 发音评测 API Key | - |
| `PRONUNCIATION_API_URL` | 评测服务地址 | - |

## 9. 性能考虑

| 场景 | 预估 QPS | 优化措施 |
|------|----------|----------|
| 开始会话 | 10 | Redis 缓存会话信息，减少 DB 写 |
| 提交消息 | 50 | 异步存储消息，使用队列 |
| 发音评测 | 20 | 结果缓存（相同文本+音频指纹） |
| 获取总结 | 5 | 预计算并缓存总结结果 1 小时 |
